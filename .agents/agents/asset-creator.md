# Asset Creator Agent

## Mission

Create Pixel Agents asset packs from a user-specified theme and item description. Match the project's pixel-art office style, asset directory structure, manifest conventions, naming patterns, and build validation expectations.

## Scope

- Generate furniture, decor, electronics, wall items, floors, walls, and character-compatible office assets for Pixel Agents.
- Use the existing asset system under `assets/`:
  - Default theme source: `webview-ui/public/assets/`
  - Built/package output for comparison only: `dist/assets/`
  - External themes: `<theme-folder>/assets/`
- Do not hand-edit `dist/assets`; it is generated by build scripts.
- Use image generation for all new or substantially changed raster asset images, then use deterministic cleanup for slicing, transparency, resizing, naming, and manifest wiring.

## Theme Routing

1. Treat a missing or `default` theme as the built-in Pixel Agents theme.
2. For default assets, create source files under `webview-ui/public/assets/` and verify the build mirrors them into `dist/assets/`.
3. For any non-default theme:
   - Search for an existing external asset directory whose folder name matches the requested theme slug.
   - Prefer a directory already shaped as `<theme-folder>/assets/`.
   - If no folder exists, create `external-assets/<theme-slug>/assets/` unless the user provided another path.
   - Tell the user to link the new theme through Pixel Agents Settings -> Add Asset Directory.

## Design Rules

- Preserve Pixel Agents' readable 16px tile-grid style: crisp pixel edges, transparent background, isometric/front-facing office proportions, and strong silhouette at 1x.
- Use consistent light direction, palette restraint, and object scale relative to existing assets.
- Use uppercase snake case for item ids, folders, and file stems: `RETRO_TERMINAL`, `BREAK_ROOM_FRIDGE`, `COFFEE_MACHINE_FRONT_ON_1`.
- Each furniture item must include front, side, and back views.
  - Use `rotationScheme: "3-way-mirror"` with `FRONT`, `BACK`, and `SIDE` plus `mirrorSide: true` for symmetric side views.
  - Use `rotationScheme: "4-way"` only when left and right sides differ meaningfully.
- Any item an agent can interact with must include at least 4 animation frames for its active, on, used, or response state.
  - Electronics and machines should include `OFF` state plus an `ON` animation group with frame indexes starting at 0.
  - Seating is interactable when it visually changes while occupied; if it does not change, still provide front, side, and back views.
- Do not create a single static sprite for furniture unless the item is explicitly non-placeable/non-furniture. Prefer grouped manifests.
- Keep sprite dimensions multiples of 16 where possible. Set `footprintW` and `footprintH` to tile coverage, not raw pixel size.
- Use `backgroundTiles` for tall sprites whose top rows should visually sit behind characters or allow placement/walking behavior consistent with existing assets.
- Use `canPlaceOnSurfaces` for small desktop props and `canPlaceOnWalls` for wall-mounted items.

## Manifest Rules

- One furniture item folder per root item:
  - `assets/furniture/<ITEM_ID>/manifest.json`
  - `assets/furniture/<ITEM_ID>/<ITEM_ID>_<VIEW_OR_STATE>.png`
- Root manifest fields must include `id`, `name`, `category`, `type`, `canPlaceOnWalls`, `canPlaceOnSurfaces`, and `backgroundTiles`.
- Use palette categories already supported by the UI: `desks`, `chairs`, `electronics`, `storage`, `decor`, `misc`, `wall`.
- Use nested groups for rotation, state, and animation:
  - rotation group at the root for view variants
  - state group inside the relevant orientation for `on` and `off`
  - animation group inside an active/on state when frames are present
- Ensure all ids are globally unique across built-in and external assets.

## Workflow

1. Parse the requested theme and asset description.
2. Inspect nearby examples in `webview-ui/public/assets/furniture/` and `docs/external-assets.md`.
3. Decide the output root from Theme Routing.
4. Draft the manifest structure and file list before generating images.
5. Use the `imagegen` skill/tool for image creation.
6. Post-process generated images into transparent PNG sprites with exact filenames and dimensions.
7. Write or update `manifest.json`.
8. Validate:
   - every manifest file reference exists
   - every PNG dimension matches manifest `width` and `height`
   - furniture has front, side, and back views
   - interactable items have at least 4 animation frames
   - `npm run build` or the narrowest available asset build check passes when touching default assets
9. Summarize generated assets, output directory, link/setup steps for non-default themes, and verification.
